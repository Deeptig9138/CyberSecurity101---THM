## 🧭 Task 1 & 2: Transport Layer Security (TLS)

Early core application protocols were built without native security controls. They transmit all transactional commands, internal application variables, and account authentication credentials across local networks in clear text.

An adversary running a packet capture utility on a network interface card configured in **promiscuous mode** can intercept all data frames passing across the shared physical medium, stripping out confidential information easily.

Furthermore, text-only communication channels fail to provide data protection across three essential pillars:

* **Confidentiality:** Preventing unauthorized third parties from reading transmitted sensitive data payloads.
* **Integrity:** Ensuring data payloads are not altered or tampered with in transit by a man-in-the-middle attacker.
* **Authenticity:** Verifying the actual identity of the target remote server to prevent interception by fraudulent look-alike endpoints.

### 🏛️ The Evolution of Network Encryption

To build security directly into existing protocols without changing the lower network layers, Netscape Communications developed **SSL (Secure Sockets Layer)**, publicly releasing SSL 2.0 in 1995. The Internet Engineering Task Force (IETF) later standardized and upgraded the framework, releasing **TLS (Transport Layer Security)** 1.0 in 1999 as an enhancement to SSL 3.0. Over two decades of ongoing security improvements culminated in the release of **TLS 1.3** in 2018, which streamlined the connection handshake and removed outdated cryptographic algorithms.

### 📜 Digital Identity Mapping via PKI

TLS relies on a Public Key Infrastructure (PKI) framework to prove identity and build encrypted sessions. The server validation workflow follows a strict three-step lifecycle:

1. **The Request:** A network administrator generates a local **CSR (Certificate Signing Request)** on their server infrastructure and submits it to a trusted third-party **CA (Certificate Authority)**.
2. **The Verification & Issuance:** The CA validates the organization's domain ownership and returns a digitally signed **TLS Certificate**.
3. **The Validation:** When a client application connects to the server, it checks the signature on the certificate against an internal store of pre-installed root certificates from trusted authorities to verify the server's identity.

> ⚠️ **Self-Signed Certificate Warning:** While administrators can generate local *self-signed certificates* to enable encryption for testing purposes, these certificates lack validation from a trusted third-party authority. Consequently, client browsers will flag them as untrusted security risks.

## 🧭 Task 3: HTTPS

**HTTPS (Hypertext Transfer Protocol Secure)** is simply standard HTTP application traffic encapsulated inside an encrypted TLS session. While unencrypted web traffic relies on TCP Port 80, secure web operations use **TCP Port 443**.

### 🔬 Network Handshake Execution Lifecycle

When a user requests a secure web document, the browser initializes a specific three-step sequence down the protocol stack after name resolution:

```text
  Client                                     Web Server
    |                                              |
    | ----- 1. TCP 3-Way Handshake ------------>   |  [Port 443]
    |                                              |
    | <---- 2. TLS Session Negotiation (8 Pkts) -> |  [Exchange Keys/Ciphers]
    |                                              |
    | ----- 3. Encrypted Application Data ------>  |  [GET /login HTTP/1.1]
    |                                              |
```

1. **TCP Three-Way Handshake:** Establishes a raw transport-layer connection channel between the client and server.
2. **TLS Session Negotiation:** The hosts exchange a series of **8 packets** to negotiate protocol versions, validate the server's digital certificate, agree on encryption algorithms (cipher suites), and securely generate shared session keys.
3. **Encrypted Application Data:** Once the secure channel is established, all high-level HTTP commands (such as a `GET /login` request) are wrapped in a protective layer, masking the plaintext payload from network eavesdroppers.

An analyst capturing traffic on an active HTTPS port without the session key will see only raw binary text labeled as **Application Data**. Once the appropriate decryption keys are loaded into a packet analyzer like Wireshark, the underlying HTTP methods, status headers, and submitted form inputs become visible for analysis.

## 🧭 Task 4: SMTPS, POP3S, and IMAPS

Just like securing web traffic, upgrading email infrastructure to protect messages and user credentials involves wrapping the underlying plaintext application commands inside a secure TLS tunnel.

Secured email protocols use dedicated port numbers to separate encrypted traffic from legacy plaintext streams:

### 📊 Email Infrastructure Protocol Port Matrix

| Application Protocol Name | Baseline Plaintext Purpose | Legacy TCP Port | Secure TLS Version | Protected TCP Port |
| --- | --- | --- | --- | --- |
| **HTTP** | Web Document Delivery | 80 | **HTTPS** | 443 |
| **SMTP** | Outbound Mail Relay | 25 | **SMTPS** | 465 / 587 |
| **POP3** | Local Mail Download & Delete | 110 | **POP3S** | 995 |
| **IMAP** | Cross-Device Mail Synchronization | 143 | **IMAPS** | 993 |

## 🧭 Task 5: SSH

The **Secure Shell (SSH)** protocol was developed in 1995 by Tatu Ylönen as an encrypted alternative to unencrypted remote management utilities like Telnet. In 1999, the OpenBSD team released **OpenSSH**, a completely open-source implementation that has become the standard utility for secure remote server administration, listening natively on **TCP Port 22**.

### 🧰 Capabilities of the OpenSSH Suite

* **Secure Authentication:** Beyond traditional password checks, SSH supports strong public-key authentication pairs and multi-factor authentication (MFA) modules.
* **End-to-End Confidentiality:** Encrypts all data passing through the connection session, protecting administrative commands from passive network interception. It also tracks remote server key fingerprints to protect users against man-in-the-middle attacks.
* **Cryptographic Integrity:** Uses cryptographic verification to ensure that data packets cannot be altered or injected in transit by an inline attacker.
* **Secure Port Tunneling:** Creates an encrypted tunnel that can route separate, unencrypted application protocols securely through the established SSH connection.
* **Graphical X11 Forwarding:** Allows administrators connecting to remote Unix hosts to run graphical user interface (GUI) applications across the network securely by passing the display data through the encrypted SSH session using the `-X` flag:

```bash
ssh 192.168.124.148 -X
```

## 🧭 Task 6: SFTP vs. FTPS

When securing file transfers, organizations deploy two distinct protocol options that use different underlying connection mechanisms:

* **SFTP (SSH File Transfer Protocol):** An extension built natively into the SSH protocol suite. It runs over **TCP Port 22**, inheriting SSH's encryption and authentication capabilities out of the box. SFTP manages all commands, directories, and file data through a single network socket, making it highly firewall-friendly.
* **FTPS (File Transfer Protocol Secure):** A modification that wraps the legacy File Transfer Protocol (FTP) inside a standard TLS session, operating by default over **TCP Port 990**. Like standard FTP, it requires managing a dual-channel architecture that uses separate network connections for commands and data transfers, which can be difficult to configure through strict firewalls.

## 🧭 Task 7: VPN

A **VPN (Virtual Private Network)** allows an organization to securely connect geographically separated offices, remote branches, and off-site employees across the public internet infrastructure.

### 🔒 The Core Properties of a VPN

* **Virtual:** It creates a logical network layout that behaves like a direct physical connection, bypassing the need to lease expensive dedicated telecommunications wiring.
* **Private:** It encapsulates all inbound and outbound network data within an encrypted **VPN Tunnel** (represented by the blue path below), protecting private corporate data as it travels over public internet routing networks.

### 🛠️ Common Deployment Implementations

* **Site-to-Site VPN:** Connects entire local area networks (LANs) across separate geographical branch offices. A hardware gateway at each site automatically encrypts and decrypts all cross-office traffic seamlessly behind the scenes.
* **Remote-Access (Client-to-Site) VPN:** Connects a single remote endpoint (such as a remote worker's laptop) directly to an internal corporate network using a software application.

### 🌐 Evasion and Traffic Routing Mechanics

When an endpoint routes its internet traffic through an active VPN tunnel, public web servers only see the external IP address of the destination VPN gateway. This mechanism helps users protect their local network location and bypass regional restrictions.

Additionally, because the local Internet Service Provider (ISP) can only see the encrypted outer tunnel traffic, they cannot read the underlying web queries or easily apply content filters.

```text
Host (Internal Private IP) -> [ Encrypted VPN Tunnel ] -> VPN Server (Japan Public IP) -> Target Web App
```

## 📝 Key Takeaways

* **Cryptographic Layer Upgrades:** TLS provides a robust framework to secure plaintext application traffic, ensuring confidentiality, data integrity, and host authenticity across unsecure networks.
* **Secure Web Lifecycle:** HTTPS establishes a TCP handshake followed by a 4-packet or 8-packet TLS negotiation process to exchange encryption ciphers before transmitting secure application data over port 443.
* **Remote Terminal Security:** SSH completely replaces legacy cleartext Telnet management systems, running over port 22 to provide secure authentication and administrative access.
* **File Security Distinction:** SFTP runs natively inside the SSH environment over port 22, whereas FTPS relies on standard TLS certificates to secure file transfers over port 990.
* **Infrastructure Tunneling:** VPN connections allow organizations to encapsulate entire network data streams inside encrypted tunnels, securely linking remote networks over public internet routing infrastructures.
