# 🗃️ Logs Fundamentals

A **log** is an immutable, timestamped digital footprint recorded automatically by an operating system, network interface, or application. Every system activity—whether a benign routine automated update or a highly malicious initial access exploitation attempt—leaves behind a distinct log stream. Tracing system events via centralized logs is the baseline requirement for modern defensive analysis.

## 🎯 Learning Objectives

By completing this room, you will learn:

* The different types of logs generated across systems and network infrastructures.
* Core principles of manual and automated log analysis techniques.
* How to navigate and filter Windows Event Logs to track malicious administrative changes.
* How to parse and interpret Web Server Access Logs to reconstruct an attacker's web activities.

## 🧭 Task 1: Introduction to Logs

Threat actors systematically attempt to clear or minimize their technical traces on a victimized system to avoid discovery. Despite these evasion tactics, security operations and digital forensics teams can successfully reconstruct the lifecycle of an intrusion, establish attack timelines, and discover the identities behind a breach.

Consider an analog crime investigation: detectives tracing the disappearance of a precious artifact from a snowy jungle cabin look for contextual evidence. They document a breached entry door, a structurally collapsed ceiling, human footprints pressed into the exterior snowy path, and footage from a neighboring residence's closed-circuit television (CCTV). Piecing together these individual traces uncovers the complete narrative.

In the digital realm, these structural traces are captured within **logs**.

### 🏢 Core Log Analytical Use Cases

| Log Operational Function | Use Case Description |
| --- | --- |
| **Security Events Monitoring** | Facilitates the real-time detection of anomalous system states, behavioral deviations, or active network intrusions before widespread compromise occurs. |
| **Incident Investigation & Forensics** | Serves as the bedrock repository for post-incident analysis. Forensics teams parse log parameters to perform root cause analysis and determine attack timelines. |
| **System Troubleshooting** | Records runtime error warnings, configuration exceptions, and application crashes to assist administrators in diagnosing or fixing software flaws. |
| **Performance Monitoring** | Tracks raw consumption metrics, throughput bandwidth, and request latencies to maintain target software health. |
| **Auditing & Compliance** | Establishes a verifiable, tamper-evident audit trail of system access configurations to satisfy compliance requirements. |

## 🧭 Task 2: Types of Logs

A major challenge in digital forensics is dealing with the sheer volume of logs. Opening an unsegmented master log file can overwhelm an analyst with thousands of simultaneous, unrelated baseline events. To prevent this, operating systems and networks split log telemetry into explicit categories based on the information they record.

### 🗄️ Standardized Infrastructure Log Categories

| Log Classification | Structural System Scope | Practical Telemetry Examples |
| --- | --- | --- |
| **System Logs** | Monitors internal operating system health, kernel status adjustments, and device engine errors. | System boot/shutdown sequences, hardware errors, device driver loading failures. |
| **Security Logs** | Captures authorization states, access policy changes, and permission boundary adjustments. | Success/failure authentication steps, user creation, account lockouts, policy overrides. |
| **Application Logs** | Documents the internal transactional state of specific software programs running on the host. | Database exceptions, software execution updates, local application internal errors. |
| **Audit Logs** | Tracks sensitive data access and structural compliance metrics across an enterprise. | Direct file modifications, database table reads, group policy updates, policy enforcements. |
| **Network Logs** | Records routing connections, transport layers, and session transitions across network segments. | Inbound/outbound traffic destinations, firewall block events, connection lengths, proxy histories. |
| **Access Logs** | Logs access records for shared web, database, or API service endpoints. | External URL access paths, raw HTTP request queries, database account connections. |

### 🔍 Log Analysis Strategy

**Log Analysis** is the systematic practice of sorting, parsing, and inspecting log events to uncover indicators of compromise (IOCs) or structural irregularities. Because scanning these massive files with the naked eye is impossible, security operations centers deploy a combination of manual command-line filtering utilities and automated log aggregation solutions.

## 🧭 Task 3: Windows Event Logs Analysis

The Windows operating system records internal security and operational states using a structured logging infrastructure. These event channels are split into three primary local log files:

* **Application Log:** Collects alerts, errors, and informational statements generated by local programs and third-party software.
* **System Log:** Tracks kernel events, service control updates, driver initialization warnings, and baseline hardware issues.
* **Security Log:** The primary focus for security reviews, tracking logon audits, access control checks, and privilege assignments.

### 🖥️ Windows Event Viewer Engine

Unlike Linux environments that store text-based log files in plain text formats, Windows uses a proprietary binary XML standard (`.evtx`). To view these files, the operating system includes a built-in graphical user interface (GUI) utility called the **Event Viewer** (`eventvwr.msc`).

The interface is broken down into three functional viewing areas:

1. **The Navigation Pane (Left):** Allows analysts to traverse local log repositories (Application, Security, System).
2. **The Event List Pane (Center):** Displays individual records sequentially, indicating the level, timestamp, source, and Event ID.
3. **The Actions Pane (Right):** Offers tools to search, export, or apply custom filters via the **Filter Current Log** feature.
   
### 🧱 Anatomy of a Windows Event Record

Double-clicking any row inside the event listing exposes the complete data structure of a logged event. Forensic investigations rely on four key structural fields:

* **Log Name:** Indicates the parent event log file tracking the event (e.g., `Security`).
* **Logged:** The precise, highly granular timestamp marking when the action hit the kernel.
* **Description:** An explicit text summary detailing the accounts, system processes, process IDs, and target variables involved in the execution.
* **Event ID:** A static, unique numerical identifier assigned by Microsoft to categorize a precise system action.

### 🔑 Critical Security Event IDs Reference

Rather than reading every unstructured sentence within descriptions, analysts filter logs by specific Event IDs to isolate critical changes instantly:

| Event ID | Standard Operational Description | Forensic Investigative Focus |
| --- | --- | --- |
| **4624** | Successful Account Logon | Establishes access timelines and maps lateral attacker movement. |
| **4625** | Failed Account Logon | Explores active password brute-forcing or dictionary attacks. |
| **4634** | Successful Account Logoff | Verifies the exact session duration of a connected profile. |
| **4720** | User Account Created | Spots attacker attempts to establish persistent local access. |
| **4722** | User Account Enabled | Flags reactivation of dormant or unauthorized accounts. |
| **4724** | Account Password Reset Attempt | Highlights attempts to hijack accounts or change credentials. |
| **4725** | User Account Disabled | Monitors denial-of-service or operational sabotage attempts. |
| **4726** | User Account Deleted | Tracks cleanup actions executed by threat actors to erase traces. |

## 🧭 Task 4: Web Server Access Logs Analysis

Every time a web browser or external client makes an HTTP request to a website (to view a page, log into an account, or upload a document), the web server records the connection entry.

For an enterprise Linux web server (such as Apache), these records are saved as plain text strings inside a dedicated log file, typically located at `/var/log/apache2/access.log`.

### 🔬 Parsing a NCSA Common/Combined Log String

A standard log entry from an Apache access log file contains specific fields formatted like this:

```text
172.16.0.1 - - [06/Jun/2024:13:58:44] "GET /products HTTP/1.1" 200 4524 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)..."
```

Forensic web analysis breaks this string down into structural indicators:

* **Client IP Address (`172.16.0.1`):** The network source address of the machine making the request.
* **Timestamp (`[06/Jun/2024:13:58:44]`):** The explicit date and time the web request hit the application.
* **HTTP Method (`GET` / `POST`):** The action requested by the client. `GET` retrieves a resource; `POST` submits data to a web form or endpoint.
* **Requested URL (`/products`):** The explicit file path or directory application requested by the user.
* **Status Code (`200`):** The server's response code. Common ranges indicate different outcomes:
* `200` $\rightarrow$ Success
* `404` $\rightarrow$ Resource Not Found (often surfaces during automated web folder directory scanning)
* `500` $\rightarrow$ Internal Server Error (can indicate application crashes caused by web exploit payloads)


* **User-Agent String:** Identifies the client operating system, hardware framework, and web browser type making the connection.

### 🛠️ Linux Command-Line Utilities for Log Auditing

Because web servers can generate millions of access log strings daily, analysts use native Linux terminal commands to process the file:

#### 1. File Consolidation (`cat`)

The `cat` command displays a log file's contents on the screen. It can also merge files that have been rotated over time into a single master log file for complete timeline analysis:

```bash
root@kali$ cat access1.log access2.log > combined_access.log
```

#### 2. String Isolation (`grep`)

The `grep` filter scans log rows and isolates lines that contain a specific keyword or IP address, stripping away irrelevant data:

```bash
root@kali$ grep "192.168.1.1" access.log
```

#### 3. Interactive Pagination (`less`)

The `less` utility allows you to scroll through massive log repositories page-by-page. It keeps the terminal responsive and uses simple navigation controls:

* **Spacebar:** Advance one page forward.
* **`b` Key:** Scroll one page backward.
* **`/` Key:** Enter search mode. Type your keyword and press **Enter**.
* **`n` Key:** Move to the next instance of the search term.
* **`N` Key:** Jump back to the previous instance of the search term.

## 🧭 Task 5: Conclusion

Logs provide the definitive evidence required to investigate security incidents and system changes. Whether analyzing Windows Event files (`.evtx`) or tracking plaintext web server access files (`access.log`), understanding log layouts helps you reconstruct an attacker's steps. Mastering these log analysis skills is essential for building robust detection rules and carrying out thorough forensic investigations.

---

## 📝 Key Takeaways

* **Immutable Evidence:** Logs serve as immutable records of activity, capturing every step of an attacker's operations across applications and networks.
* **Structural Isolation:** Segmenting log files into specific channels (System, Security, Network) prevents operational data overload during an investigation.
* **Event ID Triage:** Windows relies on specific Event IDs to identify critical security actions, such as `4624` for successful logons and `4720` for account creations.
* **Web Telemetry Parsing:** Web access logs track the source IP address, HTTP request method (`GET`/`POST`), target URL path, and the server response status code.
* **Command-Line Speed:** Native terminal tools like `cat`, `grep`, and `less` allow analysts to quickly search and filter large datasets without crashing system memory.
