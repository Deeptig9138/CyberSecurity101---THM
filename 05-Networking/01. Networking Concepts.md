## 🧭 Task 2: OSI Model

The **OSI (Open Systems Interconnection)** model is a theoretical framework developed by the International Organization for Standardization (ISO). It defines a standard stack architecture for how diverse computer systems transfer information across a network.

The OSI model breaks network tasks down into **seven distinct layers**, numbered from bottom (Layer 1) to top (Layer 7). Understanding this numbering scheme is essential for professional security triage; without it, industry terms like "Layer 3 switch" or "Layer 7 firewall" will not make contextual sense.

To easily memorize the layers from bottom to top, you can use the classic industry mnemonic:

> **P**lease **D**o **N**ot **T**hrow **S**pinach **P**izza **A**way

### 🌐 Comprehensive Breakdown of the 7 OSI Layers

![](https://github.com/Deeptig9138/CyberSecurity101---THM/blob/main/screenshots/M5/OSI-7-layers.jpg)

#### 🛑 Layer 1: Physical Layer

The Physical Layer deals exclusively with the raw hardware connections between computing endpoints. It defines the electrical, optical, or wireless signals used to transmit binary digits (0s and 1s) over physical mediums.

* **Mediums & Hardware:** Copper Ethernet cables (CAT5e/CAT6), optical fiber infrastructure, and specific wireless radio frequencies (such as the 2.4 GHz, 5 GHz, and 6 GHz Wi-Fi bands).

#### 🛑 Layer 2: Data Link Layer

The Data Link Layer establishes a local communications protocol framework between nodes sharing the same physical network segment. A network segment represents a group of local devices connected to a shared physical medium (for example, ten enterprise workstations connected to a central network switch).

* **Addressing & Standards:** Operates using **MAC (Media Access Control)** addresses, which are 6-byte (48-bit) unique hardware identifiers written in hexadecimal format (e.g., `cc:5e:f8:02:21:a7`). The first three bytes indicate the specific manufacturer vendor.
* **Protocols:** Ethernet (802.3) and Wi-Fi (802.11). Local data transmissions at this layer map a source MAC address directly to a destination MAC address.

#### 🛑 Layer 3: Network Layer

While Layer 2 manages data within a single local segment, the Network Layer handles communication across completely separate networks. Layer 3 controls logical addressing, packet segmentation, and routing decisions across the internet.

* **Core Functions:** Routers examine the logical destination IP address of incoming traffic to find the most efficient network path to forward the packets toward their destination.
* **Protocols:** Internet Protocol (IP), Internet Control Message Protocol (ICMP), and secure VPN frameworks like IPSec.

#### 🛑 Layer 4: Transport Layer

The Transport Layer manages end-to-end communication sessions directly between running host applications. It supports data flow control, packet segmentation/reassembly, and error checking to ensure data moves smoothly between processes.

* **Protocols:** Transmission Control Protocol (TCP) and User Datagram Protocol (UDP).

#### 🛑 Layer 5: Session Layer

The Session Layer is responsible for opening, managing, synchronizing, and tearing down continuous connection sessions between remote applications. It negotiates operational parameters and sets checkpoints within data streams to recover smoothly from temporary transmission drops.

* **Protocols:** Network File System (NFS) and Remote Procedure Call (RPC).

#### 🛑 Layer 6: Presentation Layer

The Presentation Layer acts as a translation layer for data. It formats, compresses, encodes, or encrypts raw application payload data to ensure that information sent from the Application layer of one system can be read cleanly by the Application layer of the receiving host.

* **Standards:** Character encodings (ASCII, Unicode), file standards (JPEG, PNG, GIF, MPEG), and MIME (Multipurpose Internet Mail Extensions), which encodes binary file attachments into 7-bit ASCII characters so they can be transmitted over email systems.

#### 🛑 Layer 7: Application Layer

The Application Layer sits at the top of the stack and provides network service interfaces directly to end-user software applications (such as a web browser or email client).

* **Protocols:** HTTP, HTTPS, FTP, DNS, SMTP, POP3, and IMAP.

## 🧭 Task 3: TCP/IP Model

Developed in the 1970s by the United States Department of Defense (DoD), the **TCP/IP model** is a practical, real-world implementation of networking standards rather than a theoretical framework. It was engineered to maintain stable network communication even during major hardware failures or physical distribution changes.

The classic TCP/IP architecture (defined in RFC 1122) groups network functions into **four core layers**. Modern educational textbooks often expand this to a **five-layer** model by separating the hardware media components into a dedicated physical layer.

## 🧭 Task 4: IP Addresses and Subnets

Every interface connected to a network requires a unique logical identifier called an **IP (Internet Protocol) address**. This address functions like a physical home postal address, ensuring that network routers can locate and deliver data packets without ambiguity.

The most common implementation is **IPv4**, which is a 32-bit logical address. These 32 bits are divided into four 8-bit sections called **octets**. Each octet can represent a decimal number ranging from `0` to `255` (yielding a theoretical maximum of roughly 4.3 billion unique combinations ($2^{32}$)).

$$\text{Example IPv4 Address Structure: } \underbrace{192}_{1\text{st Octet}} \cdot \underbrace{168}_{2\text{nd Octet}} \cdot \underbrace{66}_{3\text{rd Octet}} \cdot \underbrace{89}_{4\text{th Octet}}$$

* **Network Address:** The first address in a subnet pool (where host bits are all 0) is reserved to identify the network itself (e.g., `192.168.1.0`).
* **Broadcast Address:** The final address in a subnet pool (where host bits are all 1) is reserved to send traffic to every device on that local segment simultaneously (e.g., `192.168.1.255`).

### 💻 Auditing Network Configurations

#### Windows Terminal Execution:

```cmd
C:\> ipconfig
```

#### Linux/Unix Terminal Execution:

```bash
user@TryHackMe$ ip a s
wlo1: ... inet 192.168.66.89/24 brd 192.168.66.255
```

The shorthand notation `/24` represents the network's **subnet mask** (equivalent to `255.255.255.0`). This indicates that the leftmost 24 bits are dedicated network bits that remain constant across all local devices, leaving the remaining 8 bits to assign unique host addresses ranging from `.1` to `.254`.

### 🔐 RFC 1918 Private IP Space

To prevent global IP address depletion, standard network architectures separate the internet into public and private spaces. Private IP addresses are designed for isolated corporate or home networks and cannot route directly over the public internet. For private networks to access the internet, their edge router must translate private addresses into a public IP using **NAT (Network Address Translation)**.

Defenders must memorize the three defined **RFC 1918 private address blocks**:

* **Class A Block:** `10.0.0.0` $\rightarrow$ `10.255.255.255` (`10.0.0.0/8`)
* **Class B Block:** `172.16.0.0` $\rightarrow$ `172.31.255.255` (`172.16.0.0/12`)
* **Class C Block:** `192.168.0.0` $\rightarrow$ `192.168.255.255` (`192.168.0.0/16`)

## 🧭 Task 5: UDP and TCP

While an IP address directs traffic to a specific host on a network, the operating system needs a way to route that data to the correct application or process running on the device. This is handled at Layer 4 using **port numbers** (ranging from `1` to `65535`, with port `0` reserved).

### ⚡ UDP (User Datagram Protocol)

UDP is a minimalist, **connectionless** transport protocol. It transmits data directly to a target destination port without first verifying if the host is active or ready to receive.

* **Key Characteristics:** It does not use sequence verification, delivery confirmations, or error-recovery mechanisms. This significantly reduces network overhead, making it ideal for speed-critical services like live video streams, voice calls, and DNS queries.

### 🛡️ TCP (Transmission Control Protocol)

TCP is a **connection-oriented** transport protocol engineered for reliable data delivery. It tracks data using sequence and acknowledgment numbers to guarantee that all segments arrive intact and in the correct order.

![](https://github.com/Deeptig9138/CyberSecurity101---THM/blob/main/screenshots/M5/tcp-vs-udp-1.png)

#### 🤝 The TCP Three-Way Handshake

Before a system can send any application data over TCP, it must first establish a logical connection using a strict three-way handshake driven by control flags:

```text
  Client                               Server
    |                                    |
    | ------- 1. SYN (Seq=X) ----------> |
    |                                    |
    | <------ 2. SYN-ACK (Ack=X+1) ----- |
    |                                    |
    | ------- 3. ACK (Ack=Y+1) --------> |
    |                                    |
```

1. **SYN (Synchronize):** The client opens the connection by sending a packet with the `SYN` flag set, containing a randomly generated initial sequence number.
2. **SYN-ACK (Synchronize-Acknowledgment):** The server processes the request and responds with a packet setting both the `SYN` and `ACK` flags, acknowledging the client's sequence number and introducing its own random sequence number.
3. **ACK (Acknowledgment):** The client completes the handshake by sending a final `ACK` packet, confirming receipt of the server's response.

## 🧭 Task 6: Encapsulation

As application data travels down the network stack to be transmitted over physical media, each layer adds its own control information. This process is known as **encapsulation**. When the data reaches the receiving host, this process is reversed (**decapsulation**) as it travels up the stack to extract the raw application payload.

### 📦 Step-by-Step Data Encapsulation Lifecycle

1. **Application Data:** The process begins when a user inputs information into an application (such as typing a query into a web browser and hitting enter). The application formats this raw payload data according to its high-level protocol.
2. **Transport Layer (Segment/Datagram):** The data is passed down to Layer 4 (TCP or UDP), which appends a transport header containing the source and destination port numbers. This creates a **TCP Segment** or a **UDP Datagram**.
3. **Network Layer (Packet):** The transport unit passes down to Layer 3, which attaches an IP header containing the logical source and destination IP addresses. This creates an **IP Packet**.
4. **Link Layer (Frame):** The packet reaches Layer 2, which adds a hardware header (containing source and destination MAC addresses) and an error-checking trailer. This creates a network **Frame**, which is converted into physical signals and transmitted across the network medium.

## 🧭 Task 7: Telnet

The **TELNET (Teletype Network)** protocol is an older network standard designed for remote terminal access. Because it transmits all commands in unencrypted plain text, it is considered a major security risk and has been largely replaced by SSH for administrative access.

However, because Telnet allows users to send arbitrary text strings over a raw network connection, it serves as a highly effective utility for manually interacting with open TCP ports and probing remote services.

### 🔬 Practical Endpoint Testing via Telnet

#### 1. Testing an Echo Server (Port 7):

Connecting to a standard echo service creates an active TCP connection that mirrors back any text string you type into the console.

```bash
user@TryHackMe$ telnet 10.10.10.10 7
Trying 10.10.10.10...
Connected to 10.10.10.10.
Hi
Hi
```

*(To disconnect from an active Telnet session, press `CTRL + ]` and type `quit`).*

#### 2. Testing a Daytime Server (Port 13):

Connecting to a daytime service prompts the target host to return its current system date and timestamp before automatically closing the connection.

```bash
user@TryHackMe$ telnet 10.10.10.10 13
Connected to 10.10.10.10.
Thu Jun 20 12:36:32 PM UTC 2024
Connection closed by foreign host.
```

#### 3. Interacting with a Web Server (Port 80):

You can use Telnet to manually construct raw HTTP requests and verify web server headers by sending requests directly to port 80.

```bash
user@TryHackMe$ telnet 10.10.10.10 80
Connected to 10.10.10.10.
GET / HTTP/1.1
Host: telnet.thm
```

*(Press **Enter** twice to submit a blank trailing line, prompting the web server to process the request and return its raw response payload).*

## 📝 Key Takeaways

* **OSI vs. TCP/IP Frameworks:** The OSI model provides a 7-layer theoretical architecture, while the practical TCP/IP suite compresses these functions into a 4 or 5-layer stack.
* **Logical Triage Constraints:** Security professionals use layer designations to quickly categorize devices and systems, such as Layer 2 for MAC addressing and Layer 3 for IP routing.
* **Subnet Isolation Boundaries:** IPv4 addresses use subnet masks (like `/24`) to separate network bits from unique host addresses within a local segment.
* **Transport Reliability Differences:** TCP relies on a strict three-way handshake (`SYN` $\rightarrow$ `SYN-ACK` $\rightarrow$ `ACK`) to ensure reliable connection delivery, whereas UDP acts as a lean, connectionless alternative optimized for speed.
* **Encapsulation Flow Mechanics:** Data layers add control headers sequentially as payloads move down the stack, packaging raw application data into segments, packets, and finally frames.
