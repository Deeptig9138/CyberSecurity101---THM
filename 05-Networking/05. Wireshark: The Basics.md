# 🦈 Wireshark: The Basics

Wireshark is the industry's premier open-source, cross-platform network packet analyzer. It is capable of capturing ("sniffing") live network traffic in real time and inspecting saved packet capture files (`.pcap` / `.pcapng`). Security analysts, incident responders, and network engineers rely on Wireshark to inspect the exact bits running across the wire to troubleshoot outages or hunt for advanced threats.

## 🎯 Learning Objectives

By completing this room, you will learn:

* How to navigate and customize the Wireshark Graphical User Interface (GUI).
* How to dissect network packets and decode data across the layered TCP/IP and OSI frameworks.
* How to use packet navigation, string searching, object extraction, and expert diagnostics.
* How to build and apply basic display filters, follow application streams, and isolate network conversations.

## 🧭 Task 2: Tool Overview

### 🏢 Primary Core Use Cases

Wireshark serves as a highly versatile network diagnostic tool deployed across multiple domains:

* **Network Troubleshooting:** Identifying structural network load anomalies, bandwidth exhaustion points, and transport congestion bottlenecks.
* **Security Anomaly Detection:** Hunting for rogue host assets, mapping unauthorized port scan activity, tracking command-and-control (C2) channels, and isolating malicious traffic deviations.
* **Protocol Analysis:** Investigating low-level application parameters, reviewing response status codes, and inspecting raw data payloads.

> ⚠️ **Critical Analyst Guardrail:** Wireshark is strictly a passive **Packet Analyzer**, not an active Intrusion Detection System (IDS) or Intrusion Prevention System (IPS). It does not alter, intercept, or block network packets; it merely parses and reads them. Discovering anomalies depends entirely on the analyst's investigative skills and domain knowledge.

### 🖥️ Wireshark User Interface Blueprint

When launched, Wireshark opens into an all-in-one operational control panel broken down into five distinct primary structural zones:

* **The Toolbar:** Positioned at the absolute top, hosting main menu navigation chains alongside critical performance shortcuts to configure live sniffing, pause captures, search structures, compile summaries, and merge files.
* **The Display Filter Bar:** The central query engine panel used to type regex, syntax operators, or protocol string filters to isolate explicit traffic slices.
* **Recent Files Panel:** A quick-access inventory list archiving recently investigated capture workloads for immediate recall.
* **Capture Filter & Interfaces:** Displays available physical or software network connection boundaries (e.g., `eth0`, `lo`, `ens33`) alongside prepended capture filter bars to selectively ingest packets off the wire.
* **The Status Bar:** Sits at the absolute bottom, delivering active system status fields, active user profile maps, and total vs. displayed numeric packet tracking analytics.

### 📥 Core Data Extraction Panes

Once a capture file is fully loaded into the interface via dragging-and-dropping or opening the File menu, the display updates to reveal three highly integrated evaluation windows:

| Interface Component Pane | Structural Technical Scope | Practical Analytical Function |
| --- | --- | --- |
| **1. Packet List Pane** | Summarizes every single packet chronologically by index number, epoch arrival timestamp, source/destination address parameters, protocol types, and basic details string headers. | Serves as the master timeline matrix. Clicking any individual row locks the focus of the interface onto that specific packet. |
| **2. Packet Details Panel** | Renders a fully exploded, hierarchical, protocol-by-protocol dissection tree map of the selected packet. | Maps fields directly onto the specific layers of the OSI reference framework for targeted inspection. |
| **3. Packet Bytes Pane** | Displays the raw payload data of the packet in a side-by-side view showing raw Hexadecimal values on the left and decoded ASCII text characters on the right. | Selecting any specific field inside the Packet Details panel automatically highlights its exact underlying raw binary bytes inside this pane. |

### 🎨 Packet Colouring Engines

To accelerate human analysis, Wireshark applies default background coloring rules to rows based on protocol types and connection anomalies (e.g., failed handshakes or session drops). These coloring mechanics are divided into two operational execution modes:

* **Temporary Rules:** Applied dynamically to active workflows during a single console session using the right-click context panel or the `View --> Conversation Filter` path. These rules are cleared automatically when the file is closed.
* **Permanent Rules:** Preserved persistently across terminal restarts within the active environment profile configuration files. These rules can be built or customized using the `View --> Coloring Rules` menu window, and toggled on or off via the **Colorize Packet List** menu selector.

### 📡 Active Capture Orchestration Controls

sniffing live network interfaces depends on three primary traffic control keys on the main toolbar:

* **The Shark Fin Button (Blue):** Spins up the live capturing engine on the designated interface.
* **The Stop Button (Red square):** Pauses active packet capture loops.
* **The Restart Button (Green arrow loop):** Flushes active capture buffers and restarts the capture process on the same interface.

### 🗜️ Consolidating Captures (`Merge`)

When reviewing distributed intrusion timelines or logs from multiple sensors, analysts often need to combine multiple capture files into a single, cohesive file. Navigating through `File --> Merge` prompts you to choose a second PCAP file, displays its total packet count, and blends the streams chronologically into a unified analysis capture.

*(Note: The merged dataset must be explicitly written to disk using the Save menu before running further filter operations).*

### 📋 Viewing Capture Properties

Auditing the top-level metadata of a loaded PCAP file is essential for verifying file hashes, confirming capture timelines, and reviewing operational notes left by other engineering teams. You can open these metrics by clicking the **PCAP icon on the bottom-left corner** of the status bar or by navigating to `Statistics --> Capture File Properties`. This displays file hashes, total packet volumes, hardware interfaces, capture timelines, and custom user comments.

## 🧭 Task 3: Packet Dissection

**Packet Dissection** (or protocol dissection) is the process Wireshark uses to evaluate raw binary network data streams and decode them into human-readable fields based on standard protocol rules.

### 🔬 Multi-Layer OSI Framework Dissection Analysis

When a packet row is selected, the Packet Details panel organizes its fields into a series of drop-down menus that align directly with the structural layers of the OSI model:

```text
Packet 27 Exploded Tree Architecture:
├── Frame 27: 54 bytes on wire (Layer 1 - Physical Capture Parameters)
├── Ethernet II, Src: cc:5e:f8:02:21:a7, Dst: 44:df:65:d8:fe:6c (Layer 2 - Data Link Addressing)
├── Internet Protocol Version 4, Src: 192.168.66.89, Dst: 93.184.215.14 (Layer 3 - Network Routing)
├── Transmission Control Protocol, Src Port: 49211, Dst Port: 80 (Layer 4 - Transport Sessions)
│   └── [TCP Segments Reassembled - Layer 4 Protocol Errors / Tracking Window]
└── Hypertext Transfer Protocol (Layer 5/7 - Application Specifications)
    └── [Application Data Payload - Extensible Data Space]
```

#### 🛠️ Layer 1: The Frame (Physical Layer)

Displays hardware-level capture details, including absolute epoch arrival timestamps, interface capture indexes, frame lengths, and physical media types.

#### 🛠️ Layer 2: Ethernet II (Data Link Layer)

Exposes the physical source and destination hardware **MAC addresses**, as well as the underlying Layer 3 ethertype value (e.g., `IPv4 (0x0800)`).

#### 🛠️ Layer 3: Internet Protocol (Network Layer)

Decodes logical routing headers, showing the source and destination IP addresses, IP flags, header check sums, and the critical **TTL (Time-to-Live)** counter used to track network hops.

#### 🛠️ Layer 4: Transmission Control Protocol (Transport Layer)

Unpacks end-to-end transport session details, exposing source and destination port numbers, absolute sequence and acknowledgment tracking streams, window scale metrics, and TCP flags (such as `SYN`, `ACK`, or `FIN`).

*(Note: If a large application payload is split across multiple packets, Layer 4 extension sub-menus display reassembled TCP segment arrays and sequence analytics).*

#### 🛠️ Layer 5/7: Application Protocol Layer

Renders the high-level application protocol headers, such as raw text commands for HTTP requests/responses, FTP transfer values, or SMB file sharing details.

#### 🛠️ Application Data Extension

The bottom branch of the dissection tree displays the actual application data payload carried by the packet, such as text data, images, or eXtensible Markup Language (XML) trees.

## 🧭 Task 4: Packet Navigation

### 🔢 Unique Packet Index Tracking

Wireshark processes captured data sequentially and assigns a static, unique **Packet Number** index row to every frame. This numeric index ensures you can track specific conversations accurately and jump straight back to critical events within large capture files.

### 🔍 Advanced Text and Pattern Discovery

To find indicators of compromise (IOCs), malicious keywords, or exploit signatures within large datasets, navigate to `Edit --> Find Packet` (or press `CTRL + F`) to open the search bar utility menu.

#### ⚙️ Step 1: Input Type Selection

* **Display Filter:** Searches for packets matching explicit Wireshark filter expressions.
* **Hex Value:** Scans raw byte streams for explicit hexadecimal patterns.
* **String:** Scans the dataset for regular text keywords (supports case-insensitive matching).
* **Regex:** Evaluates the capture using complex regular expression search patterns.

#### ⚙️ Step 2: Search Pane Targeting

You must direct the search engine to look within the correct user interface window:

* **Packet List:** Searches solely within the top-level summary columns.
* **Packet Details:** Searches the exploded protocol description menus.
* **Packet Bytes:** Searches the raw hexadecimal and ASCII text payload fields.

### 📌 Operational Analysis Options

* **Marking Packets:** Right-clicking a packet and choosing **Mark/Unmark Packet** (or pressing `CTRL + M`) highlights the row in solid black text, stripping away its default protocol coloring rules. This lets you flag interesting events or isolate specific rows for export. Note that packet marks are temporary and are cleared as soon as the file session is closed.
* **Packet Comments:** You can add notes to a packet via the right-click menu or the Edit menu (`Edit --> Packet Comment`). Unlike temporary packet marks, these comments are written directly into the `.pcapng` file structure, preserving your findings for other analysts who review the file later.
* **Exporting Packets:** To share isolated slices of suspicious traffic with other teams, select the relevant rows and use the `File --> Export Specified Packets` menu to write just those rows to a new file.
* **Exporting Objects (Files):** Wireshark can automatically extract files (such as executables, images, or text documents) that were transferred over the wire during the capture session. Navigate to `File --> Export Objects` and select the relevant protocol channel (supported channels include **DICOM**, **HTTP**, **IMF**, **SMB**, and **TFTP**) to save the files to disk for deeper offline inspection.
* **Time Format Normalization:** By default, Wireshark logs time as the number of seconds elapsed since the start of the capture session. To align network events with system logs from other devices during an investigation, navigate to `View --> Time Display Format` and change the layout to **UTC Date and Time**.

### 🚨 Expert Info Diagnostics

Wireshark includes an internal diagnostic engine called **Expert Info**, which automatically flags unusual protocol states, error codes, and structural anomalies. Access this interface by clicking the **color-coded circle icon on the bottom-left** of the status bar or by navigating to `Analyze --> Expert Information`.

The engine organizes its findings into four distinct severity tiers:

* 🔵 **Chat (Blue):** Informational logs detailing standard, healthy workflow transitions (such as normal HTTP GET loops or closed TCP sessions).
* 🟢 **Note (Cyan):** Highlights unusual but acceptable events, such as application error codes or standard system warnings (e.g., HTTP `404 Not Found` responses).
* 🟡 **Warn (Yellow):** Alerts you to unexpected errors or problem statements that warrant further review, such as TCP out-of-order segments or duplicate ACKs.
* 🔴 **Error (Red):** Flags severe protocol issues or broken data packets, such as malformed packets or failed checksum calculations.

## 🧭 Task 5: Packet Filtering

### 🚦 Capture Filters vs. Display Filters

Wireshark utilizes two distinct filtering methods to manage network data:

* **Capture Filters:** Configured *before* initiating the packet capture session. They direct the driver engine to only ingest packets that match your filter rules, dropping all other traffic to save disk space and reduce processor overhead.
* **Display Filters:** Applied *after* loading a capture file. They temporarily hide irrelevant rows from view, letting you narrow your focus without altering or removing any data from the underlying file.

Defenders can quickly build display filters without memorizing syntax rules by using a simple workflow shortcut: **"If you can click on it, you can filter it."**

### 🛠️ Right-Click Filter Utilities

#### 1. Apply as Filter

Right-clicking any specific protocol field in the Details pane and choosing **Apply as Filter --> Selected** instructs Wireshark to automatically build the correct query string, insert it into the filter bar, and isolate those matching packets immediately.

#### 2. Conversation Filter

While the standard "Apply as Filter" utility isolates a single specific value, choosing **Conversation Filter** lets you isolate an entire end-to-end communication chain. You can choose to filter by Ethernet, IP, or TCP/UDP layers, automatically isolating all linked traffic between those two host sockets while hiding the rest of the noise.

#### 3. Colorize Conversation

Operates similarly to conversation filtering, but instead of hiding the surrounding traffic rows, it applies a distinct, custom background color to the entire conversation stream. This makes the conversation stand out as you scroll through the timeline window. You can reset these colors at any time by navigating to `View --> Colorize Conversation --> Reset Colorization`.

#### 4. Prepare as Filter

Builds the corresponding filter query string and places it into the display filter bar, but **waits to execute the query** until you manually press Enter. This allows you to chain multiple conditions together using logical operators like `and` or `or`.

#### 5. Apply as Column

Right-clicking a protocol field and choosing **Apply as Column** converts that specific field into a new summary column in the Packet List pane. This lets you quickly scan the value of that field across all packets in the capture timeline at a glance.

### 🔄 Stream Reconstruction (`Follow Stream`)

Network applications split files and data streams into small packets before transmitting them across the wire. To read this data clearly, analysts can reconstruct these fragments into a coherent, high-level view by right-clicking a packet row and selecting **Follow --> TCP Stream** (or UDP/HTTP Stream).

This opens a dedicated dialog box that displays the raw, reassembled application-layer conversation:

* Raw data blocks originating from the **Client** are highlighted in **Red** text.
* Raw response data returning from the **Server** are highlighted in **Blue** text.

Following a stream allows you to easily read unencrypted plain-text data, such as transmitted web content, transferred passwords, or user commands. When you close the stream window, Wireshark automatically generates and applies the correct display filter query (e.g., `tcp.stream eq 0`) to isolate just that conversation thread in the main window. Click the **"X" button** on the right side of the filter bar to clear the query and view all packets again.

## 📝 Key Takeaways

* **Passive Analysis Bounds:** Wireshark reads and parses network data passively; it does not actively block threats or alter packet data on the wire.
* **OSI Layer Alignment:** The Details pane breaks down network data into structured drop-down menus that match the exact physical, routing, and transport layers of the OSI reference model.
* **Targeted Information Discovery:** You can search for specific strings across the packet list, protocol details, or raw byte panes to quickly uncover hidden indicators and payload text.
* **Data Object Extraction:** The platform can reconstruct and extract intact files (such as images or text documents) that were transferred over network channels like HTTP or SMB.
* **Interactive Stream Reassembly:** The Follow Stream utility simplifies analysis by reassembling fragmented network packets into clean, color-coded text conversations that reflect the application-layer traffic.
