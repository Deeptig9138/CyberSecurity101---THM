## 📝 Task 1 — Introduction

This room covers three categories from the **OWASP Top 10 2025** related to application behaviour, data handling, and user-controlled input.

These vulnerabilities occur when applications:

* Store sensitive data insecurely
* Trust user input
* Accept unverified software or data
* Fail to validate integrity

The categories covered are:

1. **A04: Cryptographic Failures**
2. **A05: Injection**
3. **A08: Software or Data Integrity Failures**

### 🔐 A04: Cryptographic Failures

Cryptographic failures happen when sensitive information is not properly protected.

This can occur because of:

* Missing encryption
* Weak algorithms
* Poor key management
* Exposed secrets
* Unsafe cryptographic implementations

#### 🚨 Common Cryptographic Problems

##### Weak Encryption Algorithms

Examples:

* MD5
* SHA-1
* DES

These algorithms are outdated and vulnerable to attacks.

##### Plaintext Password Storage

Bad example:

```text
password123
```

Passwords should never be stored directly.

Instead, applications should use secure hashing algorithms:

* bcrypt
* scrypt
* Argon2

##### Hardcoded Keys

Example:

```python
SECRET_KEY="mysecret123"
```

If source code is leaked, attackers gain access to protected data.

##### Poor Key Management

Problems include:

* No key rotation
* Shared encryption keys
* Exposed secrets
* No secure storage

### 🛡️ Preventing Cryptographic Failures

Use:

* Modern encryption algorithms
* Secure hashing methods
* Proper key management systems

Examples:

* AES-GCM
* ChaCha20-Poly1305
* TLS 1.3

Store secrets using:

* AWS KMS
* Azure Key Vault
* HashiCorp Vault

Never:

* Create your own encryption algorithms
* Store secrets in code repositories

### 💉 A05: Injection

Injection occurs when an application incorrectly handles user-controlled input.

Instead of treating input as data, the application allows it to become a command or query.

### Common Injection Types

#### SQL Injection

Attackers manipulate database queries.

Example:

```sql
SELECT * FROM users WHERE username='input'
```

Unsafe input can modify the query logic.

#### Command Injection

User input reaches the operating system shell.

Example:

```text
input → system command
```

This can allow attackers to execute commands.

#### Server-Side Template Injection (SSTI)

Occurs when applications render user-controlled templates.

Attackers may execute code through template engines.

#### AI Prompt Injection

Occurs when attackers manipulate AI inputs to:

* Override instructions
* Reveal hidden information
* Alter behaviour

### 🛡️ Preventing Injection

Applications should:

* Treat all user input as untrusted
* Validate and sanitize input
* Use parameterized queries
* Avoid direct shell execution
* Escape dangerous characters

### SQL Protection

Avoid:

```text
String concatenation
```

Use:

```text
Prepared statements
Parameterized queries
```

### 📦 A08: Software or Data Integrity Failures

These failures happen when applications trust software, updates, or data without verifying authenticity.

The application assumes:

* The source is trusted
* The data is unchanged
* The software is safe

Attackers exploit this trust.

### 🚨 Common Integrity Failures

#### Unverified Updates

Applications install updates without checking:

* Signature
* Source
* Integrity

#### Unsafe External Data

Examples:

* JSON files
* Templates
* Configuration files
* Uploaded objects

#### Insecure Deserialization

Deserialization converts stored data back into objects.

If unsafe data is processed, attackers may:

* Execute code
* Modify application behaviour
* Access sensitive files

#### Supply Chain Relationship

Integrity failures are closely related to:

* Dependency attacks
* Compromised packages
* Malicious updates

### 🛡️ Preventing Integrity Failures

Implement:

* Digital signatures
* Hash verification
* Secure update processes
* Trusted sources only

For development:

* Secure CI/CD pipelines
* Protect build systems
* Validate dependencies

## 🛠 Security Best Practices

### Cryptographic Security

* Use strong encryption
* Protect keys properly
* Hash passwords securely
* Never expose secrets

### Input Security

* Validate every input
* Use safe APIs
* Avoid dynamic execution
* Escape dangerous content

### Data Integrity Security

* Verify all external data
* Validate software updates
* Secure build pipelines
* Avoid unsafe deserialization

---

## 📝 Key Takeaways

* Cryptographic failures expose sensitive information through weak protection
* Encryption must use modern, trusted algorithms
* Injection happens when applications trust user input
* SSTI can lead to server-side code execution
* Software integrity failures occur when applications trust unverified data
* Secure applications require validation at every trust boundary
