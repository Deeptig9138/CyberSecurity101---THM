## ⚙️ AS02: Security Misconfigurations

#### What Is It?

Security Misconfigurations occur when systems, applications, or infrastructure are deployed with insecure settings.

These are usually not coding mistakes but operational or deployment errors.

Examples include:

* Default credentials
* Publicly accessible storage
* Exposed administration interfaces
* Excessive permissions
* Unnecessary services

### 🌍 Real-World Example

In 2017, Uber exposed sensitive customer and driver data due to a publicly accessible AWS S3 bucket.

Attackers did not need to exploit software vulnerabilities.

They simply accessed improperly configured storage.

This demonstrates how a small configuration mistake can lead to a major breach.

### 🚨 Common Security Misconfigurations

#### Weak or Default Credentials

Examples:

```text
admin:admin
root:password
```

#### Exposed Services

Examples:

* Admin dashboards
* Development portals
* Debug interfaces

#### Misconfigured Cloud Storage

Examples:

* Public AWS S3 buckets
* Public Azure Blob Storage
* Public GCP Buckets

#### Verbose Error Messages

Error pages exposing:

* Stack traces
* File paths
* Database information
* Framework versions

#### Outdated Software

Running unsupported:

* Operating systems
* Frameworks
* Containers
* Libraries

### 🛡️ Prevention

* Remove unused services
* Change default credentials
* Apply least privilege
* Restrict network exposure
* Patch software regularly
* Review cloud permissions frequently
* Hide detailed error messages

## 📦 AS03: Software Supply Chain Failures

#### What Is It?

Software Supply Chain Failures occur when trusted third-party components introduce vulnerabilities.

Modern applications rely heavily on:

* Open-source libraries
* APIs
* Frameworks
* Containers
* AI models

A compromise in any dependency can affect every application using it.

### 🌍 Real-World Example

#### SolarWinds Attack (2021)

Attackers inserted malicious code into a legitimate software update.

Organizations installed the trusted update and unknowingly deployed malware into their environments.

The compromise occurred in the software supply chain rather than the application itself.

### 🚨 Common Supply Chain Risks

#### Vulnerable Dependencies

Using:

* Unmaintained packages
* Outdated libraries
* Unsupported software

#### Unverified Updates

Automatically trusting updates without validation.

#### Insecure CI/CD Pipelines

Attackers tamper with:

* Source code
* Build systems
* Deployment pipelines

#### AI Supply Chain Risks

Using:

* Unverified models
* Poisoned datasets
* Malicious fine-tuned models

### 🛡️ Prevention

* Verify dependencies before deployment
* Regularly patch libraries
* Audit software updates
* Secure CI/CD pipelines
* Monitor dependency vulnerabilities
* Track software provenance

## 🔐 AS04: Cryptographic Failures

#### What Is It?

Cryptographic Failures occur when encryption is implemented incorrectly or not implemented at all.

Sensitive information becomes vulnerable when:

* Weak algorithms are used
* Secrets are hardcoded
* Keys are poorly managed
* Data remains unencrypted

### Why Cryptography Matters

Cryptography protects:

* Passwords
* Session tokens
* Personal information
* API keys
* Network communications

Failures can lead to:

* Account takeover
* Data theft
* Credential exposure
* Regulatory violations

### 🚨 Common Cryptographic Failures

#### Weak Algorithms

Avoid:

* MD5
* SHA-1
* DES
* ECB Mode

#### Hardcoded Secrets

Example:

```python
SECRET_KEY = "SuperSecret123"
```

Attackers can recover secrets directly from source code.

#### Poor Key Management

Examples:

* No key rotation
* Shared secrets
* Exposed certificates

#### Missing Encryption

Sensitive data stored:

* In plaintext databases
* In logs
* In configuration files

### 🛡️ Prevention

Use modern cryptography:

* AES-GCM
* ChaCha20-Poly1305
* TLS 1.3

Manage secrets using:

* AWS KMS
* Azure Key Vault
* HashiCorp Vault

Implement:

* Key rotation
* Certificate inventory
* Secure secret management

## 🧩 AS06: Insecure Design

#### What Is It?

Insecure Design occurs when security weaknesses are built into the architecture itself.

Unlike coding vulnerabilities, insecure design originates from:

* Poor planning
* Incorrect assumptions
* Missing threat modeling
* Weak business logic

These flaws often require redesign rather than patching.

### 🌍 Real-World Example

#### Clubhouse API Exposure

The application assumed users would only interact through the mobile app.

However:

* Backend APIs lacked sufficient protections
* User data could be queried directly
* Security assumptions failed

The problem existed in the design, not the code.

### 🚨 Common Insecure Designs

#### Weak Business Logic

Examples:

* Broken approval workflows
* Flawed password recovery processes
* Missing transaction validation

#### Trusting User Behavior

Assuming users will:

* Follow intended workflows
* Not manipulate requests
* Not abuse functionality

#### AI Security Design Failures

Examples:

* Prompt injection
* Blind trust in model outputs
* Excessive AI permissions
* Unverified model behavior

#### Debug Features in Production

Examples:

* Hidden admin pages
* Test accounts
* Development bypasses

### 🤖 Insecure Design in the AI Era

Modern AI systems introduce unique risks:

#### Prompt Injection

Attackers manipulate prompts to:

* Override instructions
* Extract sensitive information
* Alter model behavior

#### Blind Trust in AI

Allowing AI-generated content to:

* Execute code
* Approve requests
* Make security decisions

without validation.

#### Poisoned Models

Models trained on malicious data may contain:

* Hidden behaviors
* Embedded backdoors
* Data leakage risks

### 🛡️ Secure Design Principles

* Threat model early and often
* Apply least privilege
* Validate all inputs and outputs
* Separate system prompts from user prompts
* Require human review for high-risk actions
* Monitor AI behavior continuously
* Define security requirements before development

## 🛠 Key Security Lessons

### Security Misconfigurations

* Never trust default settings
* Minimize exposed services
* Audit permissions regularly

### Software Supply Chain Security

* Verify dependencies
* Secure build pipelines
* Monitor third-party risks

### Cryptographic Security

* Use modern encryption
* Protect secrets properly
* Rotate keys regularly

### Secure Design

* Threat model continuously
* Validate assumptions
* Design with abuse cases in mind
* Treat AI systems as untrusted by default

* Security Misconfigurations often expose systems without requiring exploits
* Software Supply Chain attacks target trusted dependencies
* Cryptographic Failures expose sensitive information through weak protections
* Insecure Design creates vulnerabilities that cannot simply be patched
* AI introduces new architectural risks requiring dedicated threat modeling
* Secure systems begin with strong design foundations
