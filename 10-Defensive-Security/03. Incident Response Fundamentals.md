## 🧭 Task 2: What are Incidents?

Computing devices (laptops, mobile phones, servers) regularly run interactive processes (such as playing a game or watching a video) and non-interactive background processes. Every action performed by these processes generates a timestamped record known as an **event**.

Because systems run continuously, they generate a massive volume of baseline events. To manage this data, organizations ingest these events as **logs** into security solutions designed to parse and identify anomalies.

### 🚨 From Events to Alerts

When a security solution correlates an event or a group of events matching a suspected harmful activity, it triggers an **alert**. Security analysts evaluate these alerts to determine if they are legitimate or benign:

* **False Positives:** An alert that flags an operational activity as suspicious, but investigation proves it is safe.
* *Example:* A security tool flags a massive outbound data transfer to an external IP. Investigation reveals it is a scheduled system backup to a legitimate cloud storage service.


* **True Positives:** An alert that correctly identifies an active security threat or malicious behavior.
* *Example:* A security solution flags a phishing attempt. Investigation confirms the email contains a malicious link designed to compromise user credentials.



### 📉 Categorizing Incident Severity

When an alert is verified as a malicious **True Positive**, it is officially categorized as a **Security Incident**.

Because security teams often receive multiple incidents simultaneously, they use **Incident Severity** to prioritize their workload. Incidents are classified into impact tiers:

**$$\text{Low} \rightarrow \text{Medium} \rightarrow \text{High} \rightarrow \text{Critical}$$**

Critical severity incidents receive the highest operational response priority due to the severe damage or disruption they can cause to the organization.

## 🧭 Task 3: Types of Incidents

Cybersecurity incidents take many forms depending on the attacker's methods and goals. Multiple types of incidents can hit a single victim simultaneously:

* **Malware Infections:** The introduction of malicious programs (such as documents, text files, or executables) designed to compromise, disrupt, or damage endpoints, applications, or networks.
* **Security Breaches:** Incidents where an unauthorized entity successfully bypasses security controls to gain access to confidential, restricted data.
* **Data Leaks:** Occurrences where sensitive internal information is exposed to unauthorized external entities. While security breaches involve active intruders, data leaks can also be accidentally caused by internal human errors or system misconfigurations.
* **Insider Attacks:** Malicious actions intentionally executed by individuals *within* the organization (such as a disgruntled employee using a USB drive to infect the corporate network on their final day). These are highly dangerous because insiders possess legitimate credentials and higher baseline access privileges than external threat actors.
* **Denial of Service (DoS) Attacks:** Malicious campaigns aimed at disrupting system availability. Attackers flood a targeted system, network, or application with fake requests, exhausting its resources until it can no longer serve legitimate users.

> 💡 **Impact Relativity Note:** The severity of an incident type depends heavily on the victim's business model. A primary web service company might face critical financial losses from a temporary Denial of Service (DoS) attack, whereas a data leak of public information might cause minimal operational disruption.

## 🧭 Task 4: Incident Response Process

To manage security incidents effectively, organizations follow structured incident response frameworks. These frameworks provide a standardized approach to handling threats. The two industry-standard frameworks are developed by **SANS** and **NIST**.

### 🏛️ The SANS "PICERL" Framework (6 Phases)

| Phase | Explanation | Real-World Operational Example |
| --- | --- | --- |
| **1. Preparation** | Building and configuring necessary resources, training teams, establishing plans, and deploying security tools before an incident occurs. | Conducting user awareness training to help employees spot phishing emails. |
| **2. Identification** | Monitoring for abnormal system behavior and parsing security alerts to verify and document active incidents. | Noticing a massive outbound data transfer and identifying a host compromised by a phishing attachment. |
| **3. Containment** | Isolating affected systems or accounts to limit the attacker's lateral movement and minimize damage. | Disconnecting a compromised workstation from the local network. |
| **4. Eradication** | Completely removing the root cause of the threat from the environment to ensure systems are clean. | Running deep malware scans and deleting malicious files from the host. |
| **5. Recovery** | Restoring affected systems to normal operations from clean backups and testing them before return-to-service. | Re-configuring the host and restoring exfiltrated files from a secure backup. |
| **6. Lessons Learned** | Conducting post-incident reviews to document gaps, determine root causes, and update controls to prevent future attacks. | Holding a post-incident team meeting to document findings and improve detection rules. |

### 🏛️ The NIST Framework (4 Phases)

The National Institute of Standards and Technology (NIST) condenses the incident response lifecycle into **four core phases**:

1. **Preparation**
2. **Detection and Analysis** *(Corresponds to SANS Identification)*
3. **Containment, Eradication, and Recovery** *(Combines three SANS phases into a single cycle)*
4. **Post-Incident Activity** *(Corresponds to SANS Lessons Learned)*

### 📋 The Incident Response Plan (IRP)

An **Incident Response Plan** is a formal document that outlines an organization's specific procedures for managing a security incident. This document must be formally reviewed and approved by senior management. Key components of an IRP include:

* Defined roles and responsibilities for the response team.
* The chosen incident response methodology (SANS or NIST).
* Internal and external communication plans (including timelines for notifying stakeholders and law enforcement).
* Clear escalation pathways for high-severity alerts.

## 🧭 Task 5: Incident Response Techniques

Analysts use specialized security tools to automate parts of the incident response lifecycle, such as containment and eradication:

* **SIEM (Security Information and Event Management):** Collects and centralizes log data from multiple sources across the enterprise, correlating events to identify and alert on complex incidents.
* **AV (Antivirus):** Scans files local to an endpoint to detect and neutralize known malicious software signatures.
* **EDR (Endpoint Detection and Response):** Software deployed directly on endpoints to continuously monitor for advanced anomalies. EDR tools allow analysts to isolate hosts, kill malicious processes, and remediate threats remotely.

### 📖 Playbooks vs. Runbooks

To optimize response times during critical incidents, security teams rely on pre-documented technical instructions:

* **Playbooks:** High-level operational guidelines that outline the complete lifecycle response for a specific type of incident (e.g., a *Phishing Email Playbook* details steps to analyze email headers, identify recipients, isolate infected systems, and block sender domains).
* **Runbooks:** Highly detailed, step-by-step technical procedures used to execute specific tasks within a playbook based on available tools.

## 🧭 Task 7: Conclusion

Security incidents require a structured framework to minimize organizational damage and downtime. By converting raw system events into correlated alerts, defenders can catch true positive threats early. Following standard frameworks like SANS or NIST ensures teams handle everything from initial containment to post-incident lessons learned efficiently, improving security posture before the next attack occurs.

---

## 📝 Key Takeaways

* **True Positive Conversion:** Raw system logs are parsed into alerts. Legitimate security threats are categorized as true positives and tracked as formal security incidents.
* **Severity Triage:** Incidents are prioritized by severity (Low to Critical) to ensure high-impact threats are handled first.
* **Framework Alignment:** SANS uses the six-step **PICERL** framework, while NIST compresses these processes into four interconnected phases.
* **Technical Orchestration:** Security tools like SIEM aggregate log events, while EDR agents provide the endpoint visibility and control needed to isolate and clean compromised hosts.
* **Actionable Guidelines:** Playbooks provide high-level response workflows for specific threats, helping security teams reduce response times during live incidents.
