## 🧠 Core Concepts: What is IAAA?

**IAAA** is a framework used to understand how applications manage users and their actions.

Each component builds on the previous one:

### 👤 Identity

The unique account representing a person or service.

Examples:

* Username
* Email address
* Employee ID

### 🔐 Authentication

Proving ownership of an identity.

Common methods:

* Passwords
* One-Time Passwords (OTP)
* Passkeys
* Multi-Factor Authentication (MFA)

### 🛡️ Authorisation

Determines what an authenticated user is allowed to do.

Examples:

* Accessing personal data
* Viewing admin dashboards
* Editing records

### 📋 Accountability

Recording actions performed by users.

Includes:

* Logs
* Audit trails
* Alerts
* Security monitoring

### Why IAAA Matters

Failures in IAAA implementation can allow attackers to:

* Access data belonging to other users
* Escalate privileges
* Impersonate legitimate users
* Hide malicious activity

### Question

What does IAAA stand for?

### Answer

`Identity, Authentication, Authorisation, Accountability`

### 🚨 A01: Broken Access Control

Broken Access Control occurs when an application fails to properly enforce permissions.

The server should always verify:

* Who is making the request
* What they are allowed to access
* Whether the action should be permitted

When these checks are missing, users may gain access to data or functionality they should never see.

### 🔍 Insecure Direct Object References (IDOR)

A common form of Broken Access Control is:

#### IDOR (Insecure Direct Object Reference)

Example:

```url
/account?id=7
```

Changing the identifier:

```url
/account?id=6
```

If this reveals another user's data, access control is broken.

### 🔺 Privilege Escalation Types

#### Horizontal Privilege Escalation

Accessing another user's resources while remaining at the same privilege level.

Example:

* User A views User B's bank account.

#### Vertical Privilege Escalation

Accessing functionality reserved for higher privilege users.

Example:

* Normal user accessing administrator controls.

### 🏦 Practical Challenge

The challenge involves modifying the:

```url

```

parameter in the URL and identifying the user account containing more than **$1 million**.

### 🔑 A07: Authentication Failures

Authentication Failures occur when applications cannot reliably verify user identities.

These weaknesses often allow attackers to:

* Log in as other users
* Take over accounts
* Bypass authentication controls

### ⚠️ Common Authentication Weaknesses

#### Username Enumeration

The application reveals whether a username exists.

Example:

```text
Username exists
```

vs

```text
Invalid password
```

#### Weak Password Controls

Issues include:

* No password complexity requirements
* No account lockouts
* No rate limiting

This enables brute-force attacks.

#### Authentication Logic Flaws

Incorrect application logic can allow:

* Session confusion
* Account takeover
* Authentication bypasses

#### Insecure Session Handling

Examples:

* Predictable session tokens
* Session fixation
* Failure to invalidate sessions

### 🎯 Practical Challenge

The challenge demonstrates an account confusion vulnerability.

Known username:

```text

```


### 📋 A09: Logging & Alerting Failures

Logging & Alerting Failures occur when applications fail to properly record or notify security-relevant events.

Without sufficient logging:

* Security incidents go unnoticed
* Investigations become difficult
* Attack timelines cannot be reconstructed

### 🛡 Why Logging Matters

Logging provides:

* Accountability
* Auditability
* Detection capabilities
* Incident response evidence

Good logs answer:

* Who performed the action?
* When did it happen?
* Where did it originate?
* What was changed?

### 🚨 Common Logging Failures

#### Missing Authentication Events

Not recording:

* Successful logins
* Failed logins
* Password changes

#### No Brute Force Detection

Attackers can repeatedly attempt passwords without generating alerts.

#### Missing Privilege Change Logs

Role modifications and permission changes go unnoticed.

#### Short Retention Periods

Important evidence disappears before investigations begin.

#### Tamperable Logs

Logs stored locally can be modified or deleted by attackers.

### 🔍 Practical Investigation

The logs provide:

* Source IP
* Username targeted
* Endpoint accessed

This information enables defenders to reconstruct the attack timeline.

## 🛠 Key Security Lessons

### Broken Access Control

Always enforce permissions:

* On the server
* For every request
* Regardless of client-side restrictions

Never trust user-controlled parameters.

### Authentication Security

Implement:

* Strong password policies
* Multi-Factor Authentication
* Rate limiting
* Account lockouts
* Proper session management

Normalize usernames consistently to prevent account confusion issues.

### Logging & Monitoring

Ensure applications:

* Log all authentication events
* Record privilege changes
* Retain logs appropriately
* Store logs centrally
* Alert on suspicious behavior

Examples:

* Brute-force attempts
* Privilege escalation
* Multiple failed logins
* Administrative actions

## 📝 Key Takeaways

* IAAA stands for Identity, Authentication, Authorisation, and Accountability
* Broken Access Control allows unauthorized access to resources
* IDOR vulnerabilities often lead to horizontal privilege escalation
* Authentication failures can enable account takeover
* Logging is essential for detection, investigation, and accountability
* Security controls must be implemented consistently and validated server-side
